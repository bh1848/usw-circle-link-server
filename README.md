# 동구라미
> **수원대학교 동아리 통합 관리 및 지원 플랫폼** [[사이트 바로가기]](https://donggurami.net/)  
> *(현재 서비스는 종료되었습니다.)*

<br/>

## 1. 프로젝트 개요

동구라미는 수원대학교 동아리 탐색, 지원, 선발, 공지 기능을 제공하는 플랫폼입니다.  
오프라인 및 Google Form 기반 모집 절차를 앱 내 지원 시스템으로 옮겨 사용자와 운영진이 한 서비스에서 지원 및 모집 관리를 처리할 수 있도록 구성했습니다.

<br/>

## 2. 주요 기능

| 사용자 (User) | 동아리 회장 (Leader) | 중앙동아리연합회 / 운영자 (Admin) |
| :--- | :--- | :--- |
| **동아리 탐색**: 카테고리/상태별 검색 | **모집 관리**: 지원서 양식 및 모집 정보 관리 | **동아리 승인 및 운영 관리** |
| **지원**: 앱 내 지원서 제출 및 결과 확인 | **지원자 선발**: 합격/불합격 처리 | **공지사항 및 지도 관리** |
| **알림 수신**: 선발 단계별 실시간 알림 | **지원 현황 관리** | **시스템 계정 및 권한 관리** |

## 5. 담당 영역

인증/인가, 운영자/공지/지원서/동아리 기능, 공통 예외 처리, 응답 규약, 파일 업로드 검증, 입력 검증을 담당했습니다.

## 6. 트러블슈팅

### 6-1. JWT·Redis 기반 인증 구조에서 Role별 인증 주체 판별 로직을 분리해 인증 흐름 복잡도를 낮추고 유지보수성을 개선

#### 문제
서비스에는 User, Leader, Admin 세 가지 권한이 있습니다.  
권한별 로그인 대상과 접근 범위가 달라 인증 로직이 분산될 수 있었습니다.

#### 해결
Role별 로그인 흐름은 분리하고, 토큰 발급·재발급·검증 로직은 공통화했습니다.  
Refresh Token은 Redis에 저장했습니다.  
로그아웃, 재발급 실패, 비밀번호 변경 시 Refresh Token을 무효화하도록 처리했습니다.

#### 결과
- 권한별 인증 로직과 공통 인증 로직을 분리했습니다.
- Refresh Token 저장 및 무효화 기준을 Redis에서 처리했습니다.

---

### 6-2. 지원서 중복 제출 처리

#### 문제
동일 사용자가 짧은 시간 안에 중복 요청을 보내는 경우 지원서가 중복 제출될 수 있었습니다.  
지원 가능 여부 판단 로직도 서비스 내부에 함께 위치해 있었습니다.

#### 해결
지원 가능 여부 판단 로직을 분리했습니다.  
제출 시점의 상태 검증과 저장 흐름을 나누어 처리했습니다.

#### 결과
- 동일 사용자 중복 지원 가능성을 줄였습니다.
- 지원 정책과 제출 처리 로직을 분리했습니다.

---

### 6-3. 대량 삭제 및 S3 객체 정리

#### 문제
동아리, 모집글, 지원서, 첨부 파일 등 연관 데이터가 많아 개별 삭제 시 쿼리 수가 많았습니다.  
DB 데이터만 삭제되고 S3 객체가 남는 경우 고아 파일이 발생할 수 있었습니다.

#### 해결
연관 데이터 삭제에 벌크 연산을 적용했습니다.  
DB 삭제와 S3 객체 삭제를 같은 흐름에서 처리했습니다.

#### 결과
- 대량 삭제 시 쿼리 수를 줄였습니다.
- 삭제 후 남는 S3 객체를 함께 정리했습니다.

---

### 6-4. 파일 업로드 검증 강화

#### 문제
확장자만으로 파일 형식을 검증하면 위장 파일 업로드를 막기 어려웠습니다.

#### 해결
파일 확장자 검증과 함께 Magic Number 기반 시그니처 검증을 추가했습니다.  
허용한 형식과 실제 파일 시그니처가 일치하는 경우에만 업로드되도록 처리했습니다.

#### 결과
- 위장 파일 업로드를 차단했습니다.
- 파일 형식 검증 로직을 서버에서 처리했습니다.

## 7. 리팩토링

### 7-1. 응답 규약 정리

#### 기존 상태
API마다 성공/실패 응답 형태가 달랐습니다.

#### 변경
`ApiResponse`를 기준으로 응답 구조를 통일했습니다.  
전역 예외 처리기에서 공통 에러 응답을 반환하도록 정리했습니다.

#### 결과
- 응답 형식을 통일했습니다.
- 예외 응답 반환 위치를 전역 예외 처리기로 모았습니다.

---

### 7-2. 검증 레이어 분리

#### 기존 상태
입력 검증 로직이 DTO와 서비스 계층에 함께 위치할 수 있는 구조였습니다.

#### 변경
커스텀 어노테이션과 Validator로 검증 로직을 분리했습니다.  
입력 정제는 Sanitization 로직으로 별도 분리했습니다.

#### 결과
- 검증 로직과 입력 정제 로직을 분리했습니다.
- 재사용 가능한 검증 구조를 적용했습니다.
